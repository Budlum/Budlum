# Chapter 3.6: Optimistic QC and PQ Attestation

Optimistic QC is Budlum's post-quantum sidecar mechanism.

`QcBlob` contains Dilithium attestations for a finalized checkpoint:

- `epoch`
- `checkpoint_height`
- `checkpoint_hash`
- `pq_signatures`
- `merkle_root`
- `created_epoch`

Each PQ signature is produced over `epoch`, `checkpoint_height`, `checkpoint_hash`, and `validator_index`. Binding the height prevents a signature from being reused across checkpoint contexts that happen to share other fields.

QC blobs are accepted only after:

1. The checkpoint height is valid.
2. The local block hash matches `checkpoint_hash`.
3. The Merkle root matches the included signatures.
4. Every signer address and Dilithium public key matches the validator snapshot.
5. Required BLS signers are covered when finality is being accepted.

`QcFaultProof` is now available both in chain handling and as a P2P message. Peers can broadcast proof bytes, recipients parse them, verify the proof against the stored blob and historical snapshot, then apply the verdict.
